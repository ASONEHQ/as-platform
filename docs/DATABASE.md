## Scope

This document defines database philosophy and conventions. It intentionally defines no tables or product schema.

## Database philosophy

PostgreSQL is the authoritative source for transactional business data. Correctness, tenant isolation, auditability, and recoverability take priority over convenience. Redis, SQLite, IndexedDB, object storage, and future analytical projections have explicit ownership and reconciliation paths and never silently replace transactional truth.

## Naming conventions

| Object | Convention |
| --- | --- |
| Tables and columns | lowercase `snake_case` |
| Primary key | `id` |
| Foreign key | `<entity>_id` |
| Tenant ownership | `tenant_id` |
| Branch ownership | `branch_id` |
| Timestamps | descriptive `_at` suffix, stored in UTC |
| Boolean fields | positive, unambiguous names such as `is_active` |
| Constraints and indexes | stable descriptive names using repository conventions |

Avoid reserved words, unexplained abbreviations, and names tied to one customer.

## UUID strategy

Use UUIDs for primary identifiers unless an accepted ADR approves another strategy. IDs may be generated by a trusted application or database mechanism, must be globally unique, and must not encode tenant-sensitive information. A sortable UUID variant may be selected later only after compatibility and index behavior are documented.

## Tenant and branch isolation

- Every tenant-owned record has non-null `tenant_id`.
- Every branch-owned record has non-null `tenant_id` and `branch_id`.
- Foreign keys and composite constraints prevent cross-tenant or cross-branch relationships.
- Tenant-relative uniqueness includes `tenant_id`; branch-relative uniqueness also includes `branch_id`.
- Data-access interfaces require trusted scope rather than an optional tenant filter.
- PostgreSQL row-level security may provide defense in depth but does not replace application authorization and isolation tests.

## Transactions and concurrency

Sales, payments, inventory movements, cash operations, and other invariant-preserving workflows use database transactions. Keep transactions short and never hold them open across uncontrolled network calls. Use explicit locking, optimistic concurrency, or domain invariants where simultaneous updates are possible. External side effects use an outbox or resumable workflow.

## Money handling

Currency values use exact decimal or integer minor-unit representations—never floating point. Store currency where multiple currencies are possible. Define rounding, tax, allocation, and precision rules at business boundaries and test them. Corrections use explicit reversal or adjustment records.

## Record lifecycle and soft deletion

Critical transactional records are never physically deleted. Use status transitions, cancellation, reversal, or controlled soft deletion. When soft deletion is appropriate, record deletion time and actor, ensure uniqueness behavior is intentional, and exclude deleted rows through explicit repository policy. Audit evidence remains accessible to authorized roles.

## Audit logs

Critical actions record actor, tenant, branch when applicable, action, target, timestamp, correlation ID, source, reason, and safe before/after context. Audit records are append-oriented, access-controlled, protected from ordinary mutation, and subject to documented retention.

## Index strategy

Indexes follow measured access patterns. Tenant or branch scope generally leads composite indexes used by scoped queries. Foreign keys, uniqueness rules, synchronization checkpoints, and frequent ordering fields receive deliberate review. Avoid speculative indexes; assess write cost, storage, selectivity, and real query plans with representative cardinality.

## Migration strategy

- Every schema change uses a versioned migration committed with compatible application changes.
- Migrations are deterministic, reviewed, and tested on representative data.
- Deployments run migrations through one controlled job.
- Breaking or destructive changes use expand, migrate, verify, and contract stages.
- Long-running backfills are resumable, observable, and separated from blocking schema changes.
- Rollback uses a safe artifact rollback when compatible or a tested forward correction when data has changed.

## Backups and recovery

Production requires encrypted automated backups, access controls, retention schedules, integrity monitoring, and restoration procedures. Define recovery point and recovery time objectives before launch. Perform scheduled restore tests in an isolated environment and document results; an untested backup is not a recovery plan.

## Scalability considerations

Start with sound schemas, bounded queries, connection pooling, appropriate indexes, and measured query plans. Partitioning, read replicas, derived read models, archival, or specialized stores require demonstrated constraints and documented consistency behavior. Tenant isolation must survive every scaling technique.
